## 1. 引言

**1.1 目的**  
本文档旨在定义“程序请求管理模块”的功能需求。该系统将自动化并标准化作业设计程序请求的接收、分配、优先级排序、审查、批准和跟踪流程。

**1.2 目标**  
- 简化程序请求处理流程  
- 消除工程师的手动分配  
- 确保审批流程符合复杂度矩阵和服务线要求  
- 根据截止日期进行优先级排序  
- 在关键阶段通知相关人员  
- 保持所有请求的可审计性和报告能力  

**1.3 范围**  
系统支持由销售、客户或调度提交的请求，可与 FreshService 或新的自定义应用集成，按复杂度矩阵管理审批，并生成关于请求进度、分类和结果的报告。

## 2. 用户角色

**2.1 销售 / 调度**  
职责：  
- 通过邮件转发或门户表单提交程序请求  
- 附加支持文档（如示意图、许可证、客户数据）  
- 接收确认和完成通知  
- 查看程序状态以跟踪进度  
- 审查已批准的程序以用于定价或提交客户  

权限：  
- 创建请求  
- 查看自己提交请求的状态  
- 下载最终程序文档  
- 按客户、日期或程序ID搜索（仅限自己提交的）

**2.2 工程师**  
职责：  
- 接收自动分配的程序请求  
- 起草、修订并准备作业设计程序  
- 触发自动审查或将程序提交给指定审查人员  
- 处理审查/批准意见并重新提交  
- 上传最终程序文档  

权限：  
- 查看和编辑分配的请求  
- 上传和管理草稿文档  
- 查看程序的审查/批准历史  
- 搜索自己负责客户的请求

**2.3 工程经理**  
职责：  
- 为未映射工程师的请求进行手动分配  
- 监督工程师工作负载分配  
- 审查并批准黄/红色矩阵分数的作业类型  
- 确保紧急或升级请求及时处理  

权限：  
- 查看所有程序请求  
- 覆盖自动分配规则  
- 根据矩阵规则批准/拒绝请求  
- 生成工作负载和绩效报告

**2.4 总监 / 副总裁（销售与工程）**  
职责：  
- 对黑色矩阵程序提供最终审批  
- 监督审批规则的合规性  
- 审查程序绩效和审批周期指标  

权限：  
- 批准/拒绝高复杂度（黑色）请求  
- 查看完整系统报告和分析  
- 查看跨服务线的历史程序档案

**2.5 审批人员（同级/主管/经理/总监）工程师**  
职责：  
- 根据矩阵颜色审查分配的程序  
- 批准或拒绝并添加评论  
- 确保发布前符合标准  

权限：  
- 基于角色批准/拒绝请求  
- 添加审查意见  
- 查看程序生命周期和历史

**2.6 运营 / 现场团队**  
职责：  
- 获取已批准程序以执行  
- 下载最终的 PumpingPDFPrograms 用于现场作业  
- 提供反馈以持续改进  

权限：  
- 只读访问已批准请求  
- 按客户、地点或服务线搜索历史程序

**2.7 系统（Fresh Service / 自定义应用）**  
职责：  
- 基于映射自动分配请求  
- 生成程序ID  
- 在每个工作流阶段发送通知  
- 记录所有生命周期事件的时间戳  
- 记录所有操作以便审计  

权限：  
- 完全自动化权限  
- 访问所有系统数据以用于报告和分析

## 3. 高级工作流程

1. 销售/调度提交程序请求  
2. 系统记录请求并生成程序ID  
3. 请求根据公司映射自动分配给工程师  
4. 请求根据截止日期和矩阵颜色进行优先级排序  
5. 请求根据矩阵规则进行审查/批准  
6. 向销售、工程师、审批人员发送通知  
7. 批准的程序返回给销售或直接发送给客户  
8. 程序被归档并记录到数据库中

## 4. 功能需求

**4.1 请求接收/元数据字段**  
系统支持通过邮件转发或门户表单提交请求，并记录请求生成唯一程序ID。

标准字段（FreshService 已有）：  
主题、请求人、状态、优先级、分配给、创建日期、截止日期、关闭日期、审批状态、部门、来源、影响、紧急程度、类别等

所需新增字段：  
- 客户名称  
- 请求类型（如程序生成、RFP、修订、模拟、RFI、RFQ）  
- 程序类型（表层、中间层、生产层）  
- 矩阵颜色（复杂度）  
- 服务线（主作业、补救、刺激、基础设施）  
- 是否逾期  
- 预计完成时间（天数）  
- 钻机名称与编号  
- 井位/平台名称/区域名称  

字段显示自定义：  
- 用户可通过“字段选择器”自定义其任务栏中显示的字段  
- 各用户的视图可独立配置，互不影响  

**4.2 请求主题行摘要**  
系统应自动生成动态主题摘要，而非静态请求编号。摘要包括：客户名称、请求类型、服务线、唯一标识符（如LSD或平台）、是否为投标。

**4.3 工单视图配置**  
工程师打开程序请求时，系统应显示结构化布局，包含：  
- 详情  
- 相关工单  
- 任务  
- 审查  
- 批准  
- 活动  
- 解决方案（含备注）  
- 完整邮件内容（含原始消息）  
- 所有附件（邮件+文件）

**4.4 自动分配**  
系统根据公司→工程师映射表自动分配请求。  
规则：  
- 公司有映射 → 分配给对应工程师  
- 无映射 → 分配给工程经理  
- 紧急或非工作时间 → 分配给值班工程师  

通知：  
- 工程师收到邮件通知（含跳转链接）  
- 系统生成应用内通知

**4.5 优先级排序**  
系统根据截止日期和矩阵颜色自动分配优先级：  
- 紧急：4小时内截止  
- 高：当天截止  
- 中：2天内截止  
- 低：超过2天截止  

**4.6 审查**  
工程师手动将程序提交审查时，系统通知指定审查人员（邮件+应用内），含跳转链接和请求详情。

**4.7 批准**  
系统根据矩阵颜色路由程序进行审批：  
- 绿色 → 同级审查  
- 黄色 → 经理/主管审查  
- 红色 → 经理批准  
- 黑色 → 经理 + 总监/副总裁批准  

系统记录所有审批活动（时间、结果、评论），拒绝后自动退回工程师。

**4.8 多级与多作业批准**  
一个请求包含多个作业类型时，每个类型根据矩阵颜色独立审批。  
MVP阶段：最高权限者可一次性批准整个请求  
未来：支持逐个作业独立审批

**4.9 分类**  
请求提交时可分类为：  
- 投标  
- 报价  
- 已获作业  

系统支持按类别统计、筛选、报告，并展示转化率（如中标率）

**4.10 文档管理**  
支持上传或链接文档（如示意图、许可证、作业程序），记录版本历史，审批人可直接访问，最终文档保留用于审计

**4.11 状态（时间追踪）**  
系统记录每个阶段的时间戳，阶段包括：  
- 已接收  
- 已分配  
- 草稿进行中  
- 审查/批准中  
- 已批准/拒绝  
- 等待定价  
- 已执行/失标/已开票  

支持导出时间线数据用于SLA分析

**4.12 工作量估算**  
工程师可手动输入完成程序所需的工作量估算（小时或复杂度分数），未来版本将基于程序属性自动计算

**4.13 服务线区分**  
系统自动或手动标记请求所属服务线：  
- 主作业  
- 补救  
- 刺激  
- 基础设施  

决定适用的审批流程，支持按服务线筛选和报告

**4.14 步骤可视化**  
系统提供可视化进度条，显示程序当前所处阶段，用户可查看总体状态或详细步骤（含审批、评论、时间戳）

**4.15 搜索功能**  
支持多条件搜索：  
- 深度范围  
- 温度范围  
- 使用产品  
- 地理区域  
- 客户名称  
- 程序ID  
- 作业类型或矩阵颜色  

结果支持排序，响应时间要求：<3秒

## 5. 非功能需求

- 集成：FreshService API 或自定义应用  
- 可用性：自动填充字段，简洁界面  
- 性能：工单创建 <2秒，搜索 <3秒  
- 可审计性：记录所有操作、分配、审批、评论  
- 安全性：基于角色的访问控制

## 6. 报告与分析

- 平均周转时间（按公司/工程师）  
- 程序按矩阵颜色分布  
- 失标 vs 中标率  
- 工程师工作负载  
- 审批周期时长  
- 请求类型分布：投标、报价、已获作业
