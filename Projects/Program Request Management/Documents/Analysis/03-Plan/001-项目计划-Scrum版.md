# Program Request Management 项目计划 (Scrum版)

## 📋 项目概述

### 1.1 项目背景
Program Request Management (PRM) 系统旨在自动化 Program 请求管理流程，围绕自研的 PRM 应用实现从请求提交到 Program 批准的全生命周期管理。

### 1.2 Scrum转换目标
- **敏捷转型**：从瀑布式开发转向Scrum敏捷开发
- **快速交付**：通过Sprint迭代实现价值快速交付
- **持续反馈**：每个Sprint获得用户反馈，及时调整方向
- **风险控制**：通过短周期迭代降低开发风险

### 1.3 项目范围
- **核心功能**：Program 请求提交、自动分配、审批工作流、状态跟踪
- **技术栈**：.NET (ASP.NET Core + Blazor + SQL Server)
- **团队规模**：8-9人（Scrum团队）
- **迭代周期**：2周Sprint

| 角色 | 人数 | 主要职责 | Scrum参与 |

**开发团队细分：**
- 前端工程师：3人
- 后端工程师：2人  
- 全栈工程师：1人
- 测试/DevOps工程师：1人

| 活动 | 频率 | 时长 | 参与人员 |

| Sprint评审会议 | 每2周 | 2小时 | 全体团队 + 利益相关者 |

## 📈 Sprint度量指标

### 团队速率管理

**Sprint速率计算：**
- 初始预估：40-50点/Sprint（8人团队）
- 速率稳定：根据前3个Sprint实际完成情况调整
- 能力规划：基于速率调整后续Sprint范围

**关键度量：**
- **Sprint完成率**：目标 ≥ 90%
- **故事点准确度**：估算与实际偏差 ≤ 20%
- **质量门禁**：所有故事100%通过DoD
- **缺陷率**：生产缺陷 ≤ 5个/Sprint

### 燃尽图管理

**燃尽图类型：**
1. **Sprint燃尽图**：跟踪Sprint内进度
2. **发布燃尽图**：跟踪整体发布进度
3. **累积流图**：分析工作流瓶颈

**每日更新：**
- 任务剩余小时数
- 故事点完成情况
- 障碍和风险状态

## 🔄 Scrum事件详细规划

### Sprint计划会议（4小时）

**第一部分（2小时）- What：**
- Product Owner介绍Sprint目标
- 优先级最高的用户故事讲解
- 团队提问和澄清

**第二部分（2小时）- How：**
- 技术方案讨论
- 任务分解和估算
- Sprint目标确认
- 承诺最终确认

### 每日站会（15分钟）

**三问模式：**
1. 昨天完成了什么？
2. 今天计划做什么？
3. 遇到了什么障碍？

**注意事项：**
- 不是状态汇报会议
- 快速识别障碍
- 团队自组织和协作

### Sprint评审会议（2小时）

**议程安排：**
- Sprint目标回顾（15分钟）
- 功能演示（60分钟）
- 收集反馈（30分钟）
- 待办事项调整（15分钟）

**参与人员：**
- 开发团队（功能演示）
- Product Owner（需求确认）
- 利益相关者（反馈提供）
- Scrum Master（过程引导）

### Sprint回顾会议（1.5小时）

**三个问题：**
1. 做得好的地方是什么？
2. 可以改进的地方是什么？
3. 下个Sprint尝试什么改进？

**改进活动：**
- 团队满意度调查
- 流程效率分析
- 改进项跟踪

## 🎯 风险管理

### Scrum特有风险

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| Product Owner不够权威 | 高 | 明确PO职责，提供充分授权 |
| 团队技能不均衡 | 中 | 跨职能培训，技能互补 |
| Sprint范围蔓延 | 中 | 严格执行Sprint范围变更流程 |
| 利益相关者参与不足 | 中 | 建立定期沟通机制 |

### 风险应对措施

**预防措施：**
- 团队培训和能力建设
- 清晰的角色职责定义
- 定期的利益相关者沟通
- 完善的DoD和DoR标准

**应急计划：**
- 关键人员备份机制
- Sprint缓冲时间预留
- 外部专家支持计划
